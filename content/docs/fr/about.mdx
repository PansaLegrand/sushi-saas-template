## Qu’est‑ce que Sushi SaaS ?

Sushi SaaS est un starter Next.js pragmatique et orienté production pour construire et lancer de vrais produits SaaS. Il livre dès le départ l’authentification, la facturation, l’admin, le contenu et l’internationalisation, reliés à une base typée avec des migrations explicites.

Pile principale :

- Next.js App Router avec routes localisées (next‑intl)
- Better Auth + Postgres via Drizzle ORM
- Stripe Checkout + finalisation par webhooks (journal de crédits prêt)
- Docs/blogs MDX avec Fumadocs (métadonnées SEO + JSON‑LD)
- Pages d’admin avec RBAC appliqué côté serveur
- DX saine : pnpm, ESLint/Tailwind, endpoint de santé, variables d’environnement claires

Dépôt GitHub : https://github.com/PansaLegrand/saas-sushi-template

```bash
git clone https://github.com/PansaLegrand/saas-sushi-template.git
cd saas-sushi-template
pnpm install && pnpm dev
```

## Pourquoi vous pouvez lui faire confiance

- Typé de la donnée à l’UI : les schémas Drizzle sont la source de vérité ; les migrations drizzle‑kit gardent la base en phase.
- De vrais parcours, pas des stubs : auth, facturation (Stripe) et permissions admin sont connectées et exercées dans l’UI.
- i18n prévisible : routes préfixées par locale via next‑intl ; messages à côté de l’app et des docs.
- Clarté opérationnelle : `/api/health`, modèles d’environnement et consignes de build, lint et migrations.
- Documentation claire : chaque capacité (auth, billing, base, admin, e‑mails) a un guide MDX avec commandes et chemins utiles.
- Éprouvé sur le terrain : reflète des patterns utilisés pour livrer de vrais produits (voir README « Showcase »).

## Ce que vous obtenez dès le premier jour

- Authentification & RBAC : Better Auth, adaptateur BD typé, admin sécurisé (lecture seule / lecture‑écriture).
- Facturation qui tient la route : Stripe Checkout + webhooks signés pour finaliser et créditer.
- Crédits & commandes : un journal simple pour les produits à l’usage et les essais gratuits.
- Internationalisation : routes par locale, bundles de messages et docs localisées.
- Documentation MDX : Fumadocs avec navigation et table des matières.
- DX développeur : pnpm, Turbopack/Webpack, ESLint/Tailwind et conventions claires.

## Pourquoi j’ai créé ce projet open source

J’ai lancé Sushi SaaS pour rencontrer d’autres builders et me pousser à livrer régulièrement. L’open source rend cela possible : il invite de vraies conversations, déclenche des side projects et ouvre parfois la porte pour rejoindre une startup.

- Connexion et élan : je veux rencontrer des collaborateurs, explorer des side projects et—si l’alchimie opère—rejoindre une startup. Publier du code réel et fonctionnel est le meilleur déclencheur de conversation que je connaisse.
- Un meilleur logiciel grâce à l’ouverture : les boilerplates deviennent excellents quand les gens les utilisent, les critiquent et les améliorent. C’est ma façon de « build in public »—sans incitation financière—simplement par plaisir de coder et de voir d’autres construire dessus.
